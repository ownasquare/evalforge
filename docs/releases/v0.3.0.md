# EvalForge v0.3.0

EvalForge v0.3.0 connects its permanently offline workflow for human labels to completed evaluation
runs. Viewers can download a run-linked CSV template from Results and inspect immutable report
history; Editors can import completed CSV or JSON labels. API clients may request CSV or JSON
templates. EvalForge verifies the evidence without storing raw labels, result IDs, or reviewer IDs
in the application database.

## Quick start

From a source checkout with dependencies installed:

```bash
uv run evalforge demo
```

Open a completed run, turn on **Human calibration**, choose one candidate and metric, and download
the label template. Rows follow the visible case order and carry the matching case label. Fill the
`human_passed` and `reviewer_id` columns privately, select a threshold, and import the file.
EvalForge verifies the dataset, run candidate, metric version and direction, case mapping, result
IDs, and exact stored scores before it derives and saves the report. Re-importing the same manifest
and threshold returns the existing report.

The standalone `evalforge calibrate` command remains available for JSON or CSV human-label files
that should stay entirely outside the application database.

## Evidence boundaries

The stored report contains aggregate counts and quality statistics, immutable source and payload
hashes, run/candidate/metric linkage, and the creating user. It is marked
`evidence_kind: offline_statistical_evidence` and `production_validated: false`. It does not store
raw labels or reviewer identities.

Bundled fixtures and deterministic demo data are not evidence of real human review. The dashboard
workflow and `evalforge calibrate` do not approve a threshold or establish reviewer agreement. They
do not contact a model provider, certify hosted OIDC, or certify a production deployment.

EvalForge v0.3.0 is not published to PyPI or an OCI registry. The GitHub Release is the distribution
and integrity boundary for this version.
